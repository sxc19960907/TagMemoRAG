What It Does

TagMemoRAG gives teams a practical path from messy documents to usable answers: upload manuals, rebuild the index, ask a question, inspect cited sources, and review quality signals when an answer needs attention.

Why It Exists

RAG systems often demo well but become hard to operate. This project focuses on browser-first workflows, delivery readiness, safe source display, and repeatable quality checks so a real user can try the system without reading internals.

Capabilities

Everything needed for a first usable RAG experience.

Manual Library

Upload files, validate metadata, manage tags, check dirty state, and rebuild the searchable index from the browser.

User Q&A

Ask natural-language questions and receive grounded answers with citations and source cards.

Readiness Guide

See whether documents, indexing, retrieval quality, OCR, source previews, and delivery checks are ready.

Retrieval Quality

Review feedback, inspect expected evidence, and keep retrieval behavior visible during iteration.

Access Management

Manage API-key identities, scopes, KB access, and lifecycle guidance from an admin page.

Optional OCR

Use local Tesseract tooling to extract text from scanned PDF pages when native PDF text is missing.

Quick Start

The normal browser flow.

  1. Open Manual Library Create or load a knowledge base, then upload the first manual with product metadata.
  2. Rebuild the index Run a full or incremental rebuild so the document becomes searchable.
  3. Check Readiness Confirm documents, retrieval, source previews, and delivery gates before inviting users.
  4. Ask in Q&A Use the user-facing Q&A page to ask a real question and inspect citations.
  5. Review quality Use Retrieval Quality and eval reports to track weak cases and improve evidence.

Operator Guide

How to think about running TagMemoRAG.

1. Prepare Documents

Start with a small, representative set of manuals. Prefer files with clear product names, model numbers, category labels, and language metadata.

2. Build and Verify

After upload, rebuild the KB and ask questions that reflect real support tasks. Check that answers cite the right passages.

3. Add OCR When Needed

OCR is useful when a PDF page is scanned and has no native text. It feeds the same retrieval pipeline rather than becoming a separate answer path.

4. Keep Access Bounded

Give Q&A users search access, rebuild operators rebuild access, and administrators the smallest scope needed for management tasks.

5. Treat Sources as Ground Truth

Answers should be inspected through citation chips and source cards. Source previews should never expose storage keys, checksums, or local paths.

6. Ship With Readiness Checks

Before handoff, run local tests, browser flow checks, and the baseline eval suite. Use the Readiness page as a shared closure checklist.

FAQ

Common questions.

Can I read this website without installing the project?

Yes. This site is static and meant to be served publicly through GitHub Pages without running TagMemoRAG.

Does OCR require a cloud service?

No. The current OCR path is optional and uses local Tesseract command-line tooling when enabled by the operator.

Is the Q&A page the same as the admin workbench?

No. The Q&A page is user-facing and hides debug controls. Admin pages handle upload, rebuild, readiness, quality review, and access management.

What should I verify before a pilot?

Upload real documents, rebuild the KB, ask real user questions, inspect citations, confirm Readiness, and run the non-performance unit/e2e gate plus baseline eval suite.
